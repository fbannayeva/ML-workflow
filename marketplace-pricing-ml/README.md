# Marketplace Pricing ML
**End-to-end machine learning case · Dynamic pricing · Causal inference · Experimentation**

---

## Overview

This project simulates a real-world data science workflow for a ride-hailing marketplace
(inspired by platforms like Freenow, Uber, and Bolt operating in European cities).

It covers the full lifecycle of a marketplace ML product — from exploratory analysis
to model deployment to experiment evaluation — using Berlin ride data as the context.

The project is structured as four interconnected notebooks,
each building on the previous one and mirroring how data science operates in production.

---

## The Business Problem

A ride-hailing marketplace faces a core tension:

- **Too low a price** → riders convert, but revenue per ride is insufficient
- **Too high a price** → revenue per ride is high, but riders cancel or switch to competitors
- **Surge pricing** → dynamically adjusts price based on real-time supply/demand,
  maximising both driver utilisation and marketplace revenue

Beyond pricing, the marketplace needs to measure whether **incentive campaigns**
(vouchers, discounts) actually cause more rides — or just reward riders
who would have converted anyway.

This notebook builds the analytical and ML infrastructure to answer both questions —
and shows how that infrastructure moves from notebook to production.

## Notebooks

### 01 — EDA & Demand Analysis
**Goal:** Understand when, where, and at what price demand occurs —
and how supply/demand imbalances create surge pricing opportunities.

**Key findings:**
- Demand peaks at 8–9 am and 6–7 pm — classic commuter pattern
- Weekends show higher surge and higher conversion — leisure riders are less price-sensitive
- Mitte and Kreuzberg have the highest demand/supply pressure
- Price elasticity is moderate and negative — surge can be applied, but has limits

**Feeds into:** Feature selection for the pricing model in notebook 02

### 02 — Dynamic Pricing Model
**Goal:** Train an XGBoost model to predict the surge multiplier in real time,
track experiments with MLflow, package the model for production,
serve it via a FastAPI endpoint, and monitor it for data drift.

**Approach:**
- Regression task: predict `surge_multiplier` (continuous, ≥1.0)
- Features: `demand_supply_ratio`, `demand_score`, `supply_score`, `hour`, `weekday`, `is_weekend`, `zone_enc`
- Baseline: median prediction (naive benchmark)
- Model: XGBoost with regularisation (n_estimators=200, max_depth=5, lr=0.05)
- Tracking: MLflow logs all parameters, metrics, and model artifacts

**Production pipeline covered in this notebook:**
```
Train (XGBoost)
      │
      ▼
Validate (MAE, RMSE, R²)
      │
      ▼
Track (MLflow — params, metrics, artifact)
      │
      ▼
Package (model.pkl + encoder.pkl)
      │
      ▼
Serve (FastAPI — POST /predict-surge)
      │
      ▼
Monitor (KS test — daily drift detection)
      │
      ▼
Retrain (triggered when drift detected)
```

**Key results:**

| Metric | Value |
|---|---|
| Top feature | `demand_supply_ratio` |
| Inference speed | <1ms per prediction |
| Deployment | FastAPI + Docker |
| Monitoring | KS test on feature distributions |

**Real-world relevance:** In production, this model runs as a microservice
called by the pricing engine for every ride request,
receiving live supply/demand data updated every 30 seconds.

### 03 — Causal Inference: Voucher Campaign Evaluation
**Goal:** Measure the *true causal effect* of voucher discounts on conversion
using Difference-in-Differences — isolating the voucher effect from self-selection bias.

**Why not just compare voucher vs non-voucher users?**
Riders who seek vouchers are already more motivated to complete a trip.
A naive comparison overestimates the voucher effect.
DiD removes this bias by using the control group's time trend as a counterfactual.

**Method:**
```
DiD = (Treatment_post − Treatment_pre) − (Control_post − Control_pre)
    = Treatment effect  −  Natural time trend
```

**Results:**
- Parallel trends assumption validated in the pre-period ✓
- DiD estimate is positive and statistically significant (p < 0.05) ✓
- 95% bootstrap confidence interval excludes zero ✓
- Voucher ROI is positive — campaign is profitable ✓

**Business implication:** Vouchers causally increase conversions.
ROI can be further improved through uplift modelling —
targeting only price-sensitive riders who would not convert without a discount.

---

### 04 — A/B Test Evaluation
**Goal:** Design and evaluate a randomised controlled experiment testing
a 10% price reduction — the gold standard for causal inference in tech.

**Experiment design:**
- Control: standard pricing (n=400)
- Treatment: −10% price reduction (n=400)
- Primary metric: conversion rate
- Secondary metric: revenue per ride (conversion × price)
- Power analysis: ~400/group detects 5pp MDE at 80% power

**Results:**
- Statistically significant conversion lift detected ✓
- Business recommendation driven by revenue per ride, not conversion alone
- Decision framework: roll out only if revenue per ride increases

**Key principle:** Statistical significance ≠ business significance.
Always connect experiment results to P&L impact before making a rollout decision.

## Production API

After running notebook 02, a trained model is saved to `src/model.pkl`.
The FastAPI service in `src/api.py` loads it at startup and serves predictions.

**Run locally:**
```bash
pip install fastapi uvicorn
uvicorn src.api:app --reload
```

**Run with Docker:**
```bash
docker build -t pricing-api .
docker run -p 8000:8000 pricing-api
```

**Example request:**
```bash
curl -X POST "http://localhost:8000/predict-surge" \
  -H "Content-Type: application/json" \
  -d '{
    "demand_supply_ratio": 2.5,
    "demand_score": 5.0,
    "supply_score": 2.0,
    "hour": 18,
    "weekday": 4,
    "is_weekend": 0,
    "zone": "Mitte"
  }'
```

**Response:**
```json
{
  "surge_multiplier": 1.73,
  "base_price_eur": 7.0,
  "final_price_eur": 12.11
}
```

## Model Monitoring

After deployment, `src/monitor.py` runs a daily drift check
comparing the distribution of production features against the training baseline.

**Method:** Kolmogorov-Smirnov test on key features (`demand_supply_ratio`, `hour`, `zone`).

| KS p-value | Interpretation | Action |
|---|---|---|
| > 0.05 | Distribution stable | No action needed |
| < 0.05 | Drift detected | Alert team, trigger retraining |

In production this runs as a scheduled Airflow DAG.
A new model is retrained, validated, and promoted via MLflow Model Registry
before replacing the current production version.

## Relevance to Marketplace Data Science

This project directly addresses the core challenges of marketplace ML roles:

**Dynamic pricing** — real-time models that balance rider conversion and revenue per ride,
responding to live supply/demand signals.

**Causal inference** — going beyond correlation to measure whether interventions
(vouchers, discounts, promotions) actually cause the outcomes we observe.

**Experimentation** — designing statistically rigorous A/B tests and translating
results into business decisions, not just p-values.

**Production ML** — packaging models as REST APIs, containerising with Docker,
and monitoring for data drift — the gap between notebook and real business value.

**End-to-end ownership** — from raw data exploration through model training,
experiment evaluation, production deployment, and ongoing monitoring.
